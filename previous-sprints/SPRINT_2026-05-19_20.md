## Work completed

### Phase 1 — Conductor smoke test & OTel runtime fixes

Running the conductor container against the live stack revealed three failures:

**`pkg_resources` import failure (setuptools 82.x)**
- Root cause: setuptools 82 removed `pkg_resources`; older OTel packages used it at import time
- Fix: bumped all OTel packages to 1.42.0 / 0.63b0 in `microservices/conductor/requirements.txt`
  and `requirements.txt` (vtrack)

**`AsyncOpenTelemetryTransport()` TypeError**
- Root cause: API changed in 0.63b0 — the class is now a decorator wrapping an underlying
  transport, not a standalone transport
- Fix: `AsyncOpenTelemetryTransport(httpx.AsyncHTTPTransport())` in `microservices/conductor/main.py`

**`docker.errors.NotFound` crash on boot**
- Root cause: conductor tried to `get()` managed containers that had never been created
- TDD cycle: RED test (`test_skips_start/stop_when_container_not_found`) → GREEN fix with
  `try/except docker.errors.NotFound` in `_sync_start` and `_sync_stop`
- Files: `microservices/conductor/adapters/container_gateway.py`,
  `microservices/conductor/tests/test_container_gateway.py`

### Phase 2 — Grafana/Tempo "unknown_service" fix

Traces appeared in Grafana Explore but service name showed `unknown_service`.

- Root cause: `TracerProvider()` was constructed without a `Resource`. Without it OTel defaults
  the service name to `unknown_service`.
- Fix applied to all four `configure_tracing()` functions:
  ```python
  from opentelemetry.sdk.resources import Resource, SERVICE_NAME
  resource = Resource.create({SERVICE_NAME: service_name})
  provider = TracerProvider(resource=resource)
  ```
- Files: `app/tracing.py`, `microservices/conductor/adapters/tracing.py`,
  `microservices/alert-processor/tracing.py`, `microservices/notification-sender/tracing.py`

Side fix: Tempo `permission denied` on `/tmp/tempo/blocks` — named volume was root-owned.
One-time fix: `docker run --rm -v vtrack_tempo_data:/data alpine chown -R 10001:10001 /data`

### Item 2 — Prometheus custom metrics

**Baseline HTTP metrics via `prometheus-fastapi-instrumentator`**
- Added `prometheus-fastapi-instrumentator>=7.0.0` to `requirements.txt`
- Wired `Instrumentator().instrument(app).expose(app)` in `app/main.py`
- Exposes `/metrics` endpoint with standard `http_requests_total`,
  `http_request_duration_seconds_bucket` etc.
- Test suite: `TestPrometheusMetrics` in `app/tests/test_monitoring.py`

**Three custom domain metrics (`app/metrics.py` — new file)**
```python
vtrack_collection_total{slot, outcome}      # Counter — end of each guardian run
vtrack_guardian_state{slot, state}          # Gauge  — current FSM state (1=active, 0=inactive)
vtrack_collection_datapoints{slot}          # Histogram — datapoints per collection run
```

**Instrumentation sites**
- `app/scheduler.py:_run_and_record` — increments `collection_total` after each run
- `app/scheduler.py:_watch_slot._set()` — sets `guardian_state` gauge on every FSM transition;
  all non-active states set to 0.0, active state set to 1.0
- `app/scheduler.py:_watch_slot` grace_exceeded path — propagates `slot_name` to
  `collection_manager._slot` before `adapter.start()`
- `app/scraper_async.py:stop()` — observes `collection_datapoints` histogram using `self._slot`

**Test suite: `app/tests/test_prometheus_custom_metrics.py` (10 tests)**

| Class | What it verifies |
|-------|-----------------|
| `TestCollectionTotalCounter` | increments on started/missed, slot-specific labels |
| `TestGuardianStateGauge` | 1.0 on active state, 0.0 on all others after terminal |
| `TestCollectionDatapointsHistogram` | sum equals datapoints, count=1 after single stop, zero ok |

Isolation technique: unique slot name prefixes per class avoid Prometheus counter accumulation
across tests. `AsyncCollectionManager.__new__()` used to bypass `__init__` for minimal mocking
of `stop()`.

### Item 5 — Pytest collection fix for combined microservice test runs

Running `python -m pytest app/tests/ microservices/` had two problems:

**Problem 1: ProxyTracer cache invalidation**
- Root cause: `conductor.conductor._tracer` is a `ProxyTracer` that caches `_real_tracer` on
  first use. `test_conductor.py` ran before `test_conductor_otel.py` and triggered the proxy
  to bind to the app-tests provider (set by `app/tests/conftest.py:otel_exporter`). The
  conductor `_otel_provider` session fixture resets the global provider but did not clear the
  cached pointer, so conductor OTel spans were silently recorded into the wrong exporter.
- Fix: added `conductor_module._tracer._real_tracer = None` to `_otel_provider` after
  installing the in-memory provider, forcing the proxy to rebind.

**Problem 2: `tests/` namespace collision**
- Root cause: conductor and notification-sender both have `tests/__init__.py`. With pytest's
  default import mode, Python treats both as the `tests` package, causing
  `ModuleNotFoundError: No module named 'tests.test_...'` in some environments.
- Fix: added `--import-mode=importlib` to each microservice's `pytest.ini` so each test
  module is imported independently with no shared package namespace.

**Problem 3: `test_full_flow.py` collected as a test**
- Root cause: a standalone Redis integration script at the root of `notification-sender/`
  was named `test_full_flow.py` and ran top-level code at collection time (Redis connect,
  push_alert call), crashing collection with a `TypeError` once `severity` was removed.
- Fix: renamed to `manual_full_flow.py`.

**New files**
- `microservices/conductor/pytest.ini` — created (conductor had no pytest.ini)
