Migration workflow and best practices
====================================

Purpose
-------
This document explains how database schema changes are managed for this project using Alembic migrations, why we use a dedicated migration service, and how to run, test, and review migrations safely. It is written for developers of all levels; follow these steps and conventions to avoid data loss and to keep deployments repeatable and auditable.

Why Alembic?
-------------
- Versioned schema changes: Alembic stores migrations as ordered, human-readable revision files under `alembic/versions/`.
- Repeatable upgrades/downgrades: `alembic upgrade` and `alembic downgrade` let you move the schema forward or back in a controlled manner.
- Autogenerate support: `alembic revision --autogenerate` inspects SQLAlchemy models and suggests changes.

Why a dedicated `migrate` service?
----------------------------------
- Single responsibility: separates schema updates from the application process. This reduces risk of race conditions during startup.
- Same runtime as app: runs inside the same Docker image to ensure identical Python and library versions.
- Orchestrated execution: declared dependency on the database healthcheck guarantees migrations run when DB is ready.
- CI/CD friendly: you can run migrations as a CI job or Kubernetes `Job`/`initContainer`.

Files of interest
-----------------
- `alembic.ini` — Alembic configuration. The `script_location` points to `alembic/`.
- `alembic/` — Migration scripts and `env.py` used by Alembic.
- `MIGRATIONS_README.md` — This document.
- `docker-compose.yml` — Contains a `migrate` service that runs `alembic upgrade head`.

Local development: creating and applying migrations
-------------------------------------------------
1. Create a new migration (autogenerate):

```bash
# From host, using venv
/home/jc/PythonCode/vtrack/.venv/bin/alembic revision --autogenerate -m "describe change"

# Or inside the api container (recommended for parity with production):
docker compose exec api alembic revision --autogenerate -m "describe change"
```

2. Inspect the generated migration file in `alembic/versions/`. Edit it to:
- Reorder operations if needed.
- Add data migration steps for complex changes (use `op.execute()` with SQL or SQLAlchemy constructs).
- Ensure `nullable`, default, and server-side default behaviors are correct.

3. Apply the migration locally:

```bash
# Host
/home/jc/PythonCode/vtrack/.venv/bin/alembic upgrade head

# Or, using the migrate service (runs once and exits):
docker compose run --rm migrate
```

Testing migrations
------------------
- Use a disposable database instance (Docker or local) and run `alembic upgrade head` against it.
- Add a test in your CI pipeline that runs migrations against a fresh DB and runs a subset of integration tests.
- When creating migrations that alter or drop columns, add extra tests that validate data transformation logic.

Rollback and downgrades
-----------------------
- Downgrades are not always implemented automatically; review each `down_revision` section.
- For destructive changes (dropping columns/tables), prefer forward-compatible deploys: add new columns, backfill, switch reads/writes, then remove old columns in a later migration.
- Always take backups before running destructive migrations in production.

Best practices and conventions
------------------------------
- Revision messages: use clear, imperative messages like "Add student_status_ts nullable".
- One logical change per migration: keep revisions focused to make reviews easier.
- Review autogenerated migrations: never blindly apply them in production without review.
- Avoid large data migrations in the same transaction as schema changes for very large tables—use separate scripts or background jobs.
- Use `alembic` in CI to ensure migrations are reproducible and to detect issues early.

Running migrations in CI/CD
---------------------------
- Add a job that runs `docker compose run --rm migrate` or runs Alembic inside your build image before service deployment.
- Ensure the CI runner has access to the database or a staging database and that migrations run against the correct `DATABASE_URL`.

Operational safety
------------------
- Back up the DB before production migrations.
- Run migrations during a maintenance window if you anticipate long-running schema changes.
- Monitor logs for migration failures and keep migration output accessible (container logs).

Troubleshooting
---------------
- "No 'script_location' key" — ensure `alembic.ini` is present and `script_location` is set. The repository copies `alembic.ini` into the container at `/app/alembic.ini`.
- Authentication errors — verify `DATABASE_URL` in `.env` and that the container can reach the DB service (`db`).
- Migration missing from container — if `alembic/` or `alembic.ini` are not present in the image, rebuild the `api` image (`docker compose build api --no-cache`).
